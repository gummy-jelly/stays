# JMeter Summary Report 분석 프롬프트

아래 프롬프트를 ChatGPT 또는 Claude에 붙여넣고, [여기에 붙여넣기] 부분에 JMeter 결과를 넣으면 됩니다.

---

## 프롬프트

```
당신은 성능 테스트 전문가입니다.
아래는 JMeter로 숙박 예약 서비스에 부하테스트를 진행한 Summary Report 결과입니다.

**테스트 환경:**
- 대상 서버: FastAPI 백엔드 (단일 서버, 온프레미스)
- DB: PostgreSQL (SELECT FOR UPDATE 락 사용)
- 시나리오: 1000명이 동시에 쿠폰 선착순 발급 요청
- Ramp-up: 10초 동안 1000명 점진적 증가
- 쿠폰 재고: 500장 (선착순 500명만 성공)

**JMeter Summary Report 결과:**
[여기에 붙여넣기]

(예시 형식)
Label                     | # Samples | Average | Min | Max  | Std. Dev. | Error % | Throughput | Received KB/s
① POST /auth/login        |      1000 |    1234 |  45 | 8921 |       987 |   2.30% |      89.5  |  23.4
② GET /events/:id         |       970 |     456 |  32 | 3210 |       234 |   0.00% |      87.2  |  45.6
③ POST /coupons/issue ★  |       970 |    5823 |  89 | 30000|      4521 |  48.45% |      45.1  |  12.3
TOTAL                     |      2940 |    2504 |  32 | 30000|      3102 |  17.21% |     221.8  |  81.3

**분석해주세요:**

1. **전체 성능 평가** - 이 결과가 정상인지 비정상인지, 어떤 수준인지 평가해주세요.

2. **병목 구간 식별** - 어떤 API에서 가장 심각한 문제가 발생했나요? 원인은 무엇인가요?

3. **Error % 해석** - 에러율이 높은 이유가 서버 문제인지, 쿠폰 소진으로 인한 정상적인 실패인지 판단해주세요.

4. **응답시간 분석** - Average, Max, Std. Dev.를 기반으로 응답시간 안정성을 평가해주세요.

5. **Throughput 분석** - 현재 서버가 초당 처리할 수 있는 요청 수와 한계를 설명해주세요.

6. **DB 락 영향** - SELECT FOR UPDATE 락으로 인한 병목이 결과에 나타나고 있나요?

7. **스케일아웃 필요성** - 이 결과를 근거로 AWS 오토스케일링이 왜 필요한지 설명해주세요.

8. **개선 방안 제안** - 성능 개선을 위한 방안을 3가지 제안해주세요. (스케일아웃 외에도)
```

---

## 사용 방법

1. JMeter GUI → Summary Report 탭 클릭
2. 결과 테이블을 텍스트로 복사 (또는 스크린샷)
3. 위 프롬프트의 `[여기에 붙여넣기]` 부분에 결과 붙여넣기
4. ChatGPT / Claude에 전송

---

## CLI 결과 붙여넣는 경우

JMeter CLI 실행 시 터미널에 출력되는 요약도 사용 가능:

```
summary = 3000 in 00:01:32 = 32.6/s Err: 485 (16.17%)
```

이 경우 프롬프트에 추가:
```
CLI 실행 요약:
summary = [여기에 붙여넣기]
```

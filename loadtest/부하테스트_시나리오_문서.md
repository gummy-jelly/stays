# Stays 서비스 부하 테스트 시나리오 문서

## 개요

온프레미스 서버 CPU 과부하 → AWS 자동 확장 트리거를 시연하기 위한 부하 테스트.
핵심 병목 지점은 쿠폰 발급(`POST /coupons/issue`)과 예약 생성(`POST /bookings`)의
PostgreSQL `SELECT FOR UPDATE` 비관적 락으로 인한 DB 락 경합이다.

- **도구**: Apache JMeter 5.6+ / BlazeMeter
- **대상 서버**: VM2 (API 서버) `http://VM2_IP:8000`
- **사전 준비**: `scripts/create_users.py` 실행 → `scripts/users.csv` 생성

---

## 사전 준비

### 1. 테스트 유저 생성
```bash
cd loadtest/scripts
pip install requests

loadtest/results/ 폴더가 있는지 확인
```

## 시나리오 1 — 쿠폰 선착순 폭발

**파일**: `jmeter/scenario1_coupon_rush.jmx`

### 상황
이벤트 오픈 공지 직후, 1000명이 동시에 쿠폰 선착순 발급 요청.
쿠폰은 500장 한정 → 절반만 성공, 나머지는 "소진됨" 응답.

### 부하 설정
| 항목 | 값 |
|---|---|
| 동시 유저 수 | 1,000명 |
| Ramp-up | 10초 (순간 폭발) |
| 반복 | 1회 |

### API 흐름
```
① POST /auth/login          로그인 (CSV 계정)
② GET  /events/:id          이벤트 상세 확인
③ POST /coupons/issue  ★   쿠폰 발급 요청 (DB 락 핵심)
```

### 예상 결과
- `POST /coupons/issue` 구간에서 CPU 급격히 상승
- 500명 성공 (HTTP 200) / 500명 실패 (HTTP 409 "소진됨")
- **AWS 자동 확장 트리거 1순위**

## 시나리오 2 — 특가 숙소 예약 경쟁

**파일**: `jmeter/scenario2_booking_race.jmx`

### 상황
특가 숙소 객실 마감 임박 공지 후, 500명이 남은 객실 선점 경쟁.
객실 수 30개 → 30명만 예약 성공.

### 부하 설정
| 항목 | 값 |
|---|---|
| 동시 유저 수 | 500명 |
| Ramp-up | 5초 (매우 빠르게) |
| 반복 | 1회 |

### API 흐름
```
① POST /auth/login            로그인
② GET  /stays/:id             숙소 상세 확인 (Think time 1초)
③ GET  /stays/:id/rooms       객실 잔여 확인
④ POST /coupons/validate      쿠폰 유효성 체크
⑤ POST /bookings         ★   예약 생성 (remaining_count 경합)
```

### 예상 결과
- `POST /bookings` 구간에서 DB 락 집중
- 30명 성공 (HTTP 201) / 470명 실패 (HTTP 409 "객실 없음")
- 시나리오1 직후 실행 시 CPU 누적 상승

## 시나리오 3 — 이벤트 오픈 복합 플로우

**파일**: `jmeter/scenario3_event_open.jmx`

### 상황
이벤트 오픈 후 800명이 실제 사용자처럼 이벤트 확인 → 쿠폰 발급 → 숙소 탐색 → 예약까지 전체 플로우 진행. 가장 현실적인 시나리오.

### 부하 설정
| 항목 | 값 |
|---|---|
| 동시 유저 수 | 800명 |
| Ramp-up | 30초 (현실적 유입 속도) |
| 반복 | 1회 |

### API 흐름
```
① POST /auth/login            로그인
② GET  /events                이벤트 목록 탐색
③ GET  /events/:id            이벤트 상세 확인 (Think time 2초)
④ POST /coupons/issue    ★   쿠폰 발급 (DB 락 1차)
⑤ GET  /stays/:id            숙소 탐색 (Think time 2초)
⑥ GET  /stays/:id/rooms      객실 선택
⑦ POST /bookings         ★   예약 생성 (DB 락 2차)
```

### 예상 결과
- 쿠폰 + 예약 두 지점에서 DB 락 복합 발생
- CPU가 두 번 스파이크 형태로 상승
- **BlazeMeter 시연용으로 가장 적합**

## 시나리오 4 — 인기 숙소 집중 조회

**파일**: `jmeter/scenario4_read_flood.jmx`

### 상황
SNS 바이럴로 특정 숙소에 조회 트래픽 폭증. DB 쓰기 락 없이 읽기 부하만 발생.
다른 시나리오와 CPU 패턴 비교용.

### 부하 설정
| 항목 | 값 |
|---|---|
| 동시 유저 수 | 2,000명 |
| Ramp-up | 60초 (서서히 증가) |
| 반복 | 3회 (지속적 조회) |

### API 흐름
```
① GET /stays                  숙소 목록 조회
② GET /stays/:id              특정 숙소 상세 (반복 집중)
③ GET /stays/:id/rooms        객실 정보 조회
```

### 예상 결과
- DB 쓰기 락 없음 → 상대적으로 완만한 CPU 상승
- 시나리오1~3 대비 CPU 차이로 락 영향도 확인
- 2000명에서도 응답시간 안정적이면 읽기 성능 양호

## 시연 실행 순서

```
Step 1  시나리오4 실행 (조회 폭주)
        → CPU 완만하게 상승, 정상 범위 유지 확인

Step 2  시나리오1 실행 (쿠폰 선착순)
        → CPU 급등 → AWS 자동 확장 트리거 발동

Step 3  시나리오3 실행 (복합 플로우)
        → 쿠폰+예약 두 단계 DB 락, CPU 지속 부하

Step 4  시나리오2 실행 (예약 경쟁)
        → CPU 누적 상승, AWS 추가 자원 투입 확인
```

## JMeter 실행 방법

### 로컬 실행 (GUI — 테스트 확인용)
```bash
jmeter -t jmeter/scenario1_coupon_rush.jmx
```

### 로컬 실행 (CLI — 실제 부하용)
```bash
# BASE_URL과 API_PORT를 실제 서버 주소로 지정
jmeter -n \
  -t jmeter/scenario1_coupon_rush.jmx \
  -JBASE_URL=http://VM2_IP \
  -JAPI_PORT=8000 \
  -l results/scenario1_result.jtl \
  -e -o results/scenario1_report
```

### BlazeMeter 업로드 방법
1. [blazemeter.com](https://blazemeter.com) 접속 → Create Test → JMeter
2. `.jmx` 파일 업로드
3. User Defined Variables에서 `BASE_URL`을 VM2 IP로 수정
4. Load Configuration 설정 (Users / Ramp-up은 JMX 값 그대로 사용)
5. Run Test
